# v1.1.0 - 集成 LlamaParse 云端智能解析，升级文档提取能力

## 背景与问题分析

### 现有方案的瓶颈
当前 `extractor.js` 使用的解析库存在以下明确的已知缺陷：

| 文件类型 | 使用库 | 已知缺陷 |
|---------|------|---------|
| `.pdf` | `pdf-parse` | 1. 扫描版/图片型PDF完全无法提取文字（输出为空）<br>2. 多栏排版会把两栏文字混合成乱序文字流<br>3. 表格内容会被提取成一行乱串的文字，结构丢失 |
| `.docx` | `mammoth` | 仅提取纯文字，表格会丢失行列结构，复杂格式（如分栏、文字框）会被跳过 |

上述问题在 RAG（检索增强生成）中的危害尤为严重——大模型接收到"乱掉"的表格文字或乱序文字流，其检索和回答质量会大幅下降。

---

## 解决方案设计

### 方案选择：混合解析架构（Cloud-First with Local Fallback）

采用 **"优先调用 LlamaParse 云端解析 + 本地库兜底"** 的双层架构：

```
用户上传文件 (PDF/DOCX)
        │
        ▼
[检查 config.json 中是否有 llamaParse.apiKey?]
        │
   ┌────┴────┐
   │  有 Key  │                    │  没有 Key  │
   ▼          │                    ▼
[调用 LlamaParse API]          [使用本地 pdf-parse / mammoth]
[获取结构化 Markdown]           [降级处理，输出纯文本]
   │
   ▼
[cleanText() 做最终清洗]
   │
   ▼
[存储到数据库，进行 RAG 向量化]
```

### 为什么选择 LlamaParse？
- **专为 RAG 设计**：直接输出 LLM-ready 的 Markdown 格式，表格、标题层级完整保留
- **内置 OCR**：扫描版 PDF（图片型）也能提取文字
- **免费额度充足**：Free Plan 提供 10,000 次/月 解析额度
- **接入成本极低**：只需一个 API Key，通过 HTTP multipart POST 上传文件即可

## 技术实现规范

### 1. `config.json` 新增字段

```json
{
  "llamaParse": {
    "apiKey": "llx-xxxx",
    "baseURL": "https://api.cloud.llamaindex.ai/api/parsing",
    "language": "ch_sim",
    "resultType": "markdown"
  }
}
```

- `language`: 指定 OCR 识别的主语言，`ch_sim` 为简体中文。
- `resultType`: 固定为 `"markdown"`，让 LlamaParse 输出结构化的 Markdown 格式。

### 2. LlamaParse API 调用流程（两步式异步）

LlamaParse 不是同步返回，而是采用 **"上传 → 轮询等待 → 拉取结果"** 的异步模式：

```
步骤 1: POST /upload
  Body: form-data { file: <文件二进制流> }
  Response: { id: "job-uuid-xxx", status: "PENDING" }

步骤 2: GET /job/{id}  (轮询，每2秒一次)
  Response: { status: "SUCCESS" | "PENDING" | "ERROR" }

步骤 3: GET /job/{id}/result/markdown
  Response: { markdown: "# 标题\n\n| 列1 | 列2 |\n..." }
```

### 3. `extractor.js` 改造点

- 新增函数 `extractWithLlamaParse(filePath)`: 调用云端 API
  - 读取本地文件为 `Buffer`
  - 使用 `FormData` 构建 multipart 请求
  - 轮询直到 `status === 'SUCCESS'`，最多等待 300 秒（60 次 × 5 秒）
  - 拉取 Markdown 格式的结果并返回
- 修改 `extractText(filePath)` 主函数：
  - 对 `.pdf` 和 `.docx` 文件，**优先**调用 `extractWithLlamaParse()`
  - 如果 LlamaParse 不可用（无 API Key 或网络异常），**自动 fallback** 到原有的 `pdf-parse` / `mammoth`
  - `.txt` 和 `.md` 文件仍直接本地读取（无需云端解析）

### 4. 依赖变更

本次改造**不引入任何新的 npm 包**。`fetch` 在 Node.js 18+ 中已内置，可以直接调用 HTTP 接口。

> ⚠️ 需要确认 Electron 使用的 Node.js 版本 ≥ 18，以使用内置 `fetch` 和 `FormData`。
