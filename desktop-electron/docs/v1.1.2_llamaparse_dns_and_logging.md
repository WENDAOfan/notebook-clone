# v1.1.2 - 解决 LlamaParse 间歇性 "fetch failed"：DNS IPv4 优先与详细错误日志

## 问题背景与分析

虽然我们在 `v1.1.1` 中关闭了 Keep-Alive（添加了 `Connection: close`），但用户在重新运行 `npm start` 后，在上传文件时依然会遇到 `fetch failed` 错误。

### 根源分析
在 Node.js 17+ (包括 Electron v28 使用的 Node.js 18.18.2) 中，DNS 解析默认会根据操作系统的配置顺序进行。如果用户的网络环境（例如部分中国电信/联通网络）配置了不稳定的 IPv6 地址，Node.js 会优先尝试连接 LlamaParse 服务器的 IPv6 地址。
- **现象**：由于 IPv6 路由不通或极慢，请求在握手阶段就会超时，最终抛出 `fetch failed`。
- **间歇性**：有时 DNS 返回 IPv4，上传就能成功；有时返回 IPv6，上传就直接报错。

此外，当前 `extractor.js` 中的错误捕获仅打印了极其模糊的 `err.message`（即 `fetch failed`），把底层的真实原因（如 `ETIMEDOUT` 或 `ECONNREFUSED`）隐藏了，给排查带来了困难。

---

## 解决方案设计

### 1. 强制全局 DNS 优先解析 IPv4
在 Electron 的主进程入口 `main.js` 最顶部引入 `dns` 模块，并设置默认解析顺序为 `ipv4first`：
```javascript
const dns = require('dns');
if (typeof dns.setDefaultResultOrder === 'function') {
  dns.setDefaultResultOrder('ipv4first');
}
```
这能强制 Node.js 优先使用成熟稳定的 IPv4 进行网络连接，彻底解决因 IPv6 握手失败造成的 `fetch failed` 报错。

### 2. 增强 `extractor.js` 异常日志输出
改造 `extractor.js` 中的错误捕获逻辑，在 `console.warn` 中打印出 `err.cause`（包含真实的 Node 网络连接错误堆栈），以便于精确定位底层问题：
```javascript
} catch (err) {
  console.warn(`[Extractor] LlamaParse 解析失败，降级至本地（LiteParse/mammoth）。原因: ${err.message}`, err.cause || '');
  // 继续执行下方的 fallback 逻辑
}
```
