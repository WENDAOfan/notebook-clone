# v1.1.1 - 修复 LlamaParse 轮询期间 "fetch failed" 连接中断问题

## 问题现象与原因分析

在 Electron 主进程运行 `npm start` 导入 PDF 时，控制台打印出：
```
[LlamaParse] 上传成功，Job ID: 5579a4f0-a9d7-4b28-b0ff-17bd2a2f7207，开始轮询...
[LlamaParse] 轮询 1/60，状态: PENDING
[Extractor] LlamaParse 解析失败，降级至本地（LiteParse/mammoth）。原因: fetch failed
```

### 根源分析
1. **Node.js Keep-Alive 预连接超时复用 Bug (undici)**：
   Node.js 18+ 内置的 `fetch`（由 `undici` 提供支持）默认开启了 HTTP Keep-Alive 连接保持。在第一次轮询请求成功后，undici 会试图复用当前的 TCP 连接套接字进行下一次请求。
2. **间隔断开**：
   两次轮询之间有 5 秒的 `sleep` 间隔。当网络不够稳定或 LlamaParse 远程服务器在 5 秒内单方面关闭/断开（FIN）了该闲置连接时，Node.js 依然向该失效 socket 发送第二次请求，从而引发 `ECONNRESET` 并被 Node.js 包装成 `TypeError: fetch failed` 抛出。

---

## 解决方案设计

我们将通过向 LlamaParse API 的所有 `fetch` 请求中加入 `'Connection': 'close'` 请求头，强制在每次 HTTP 事务结束后关闭连接，避免 undici 的 Socket 复用机制导致连接中断。

### 接口修改意图

在 `extractor.js` 中的 3 处 `fetch` 接口中，增加 `'Connection': 'close'`：

1. **上传请求 (`POST /upload`)**
   ```javascript
   const uploadRes = await fetch(`${baseURL}/upload`, {
     method: 'POST',
     headers: {
       'Authorization': `Bearer ${apiKey}`,
       'Connection': 'close' // 禁用 Keep-Alive
     },
     body: formData,
   });
   ```

2. **轮询状态请求 (`GET /job/{id}`)**
   ```javascript
   const statusRes = await fetch(`${baseURL}/job/${jobId}`, {
     headers: { 
       'Authorization': `Bearer ${apiKey}`,
       'Connection': 'close' // 禁用 Keep-Alive
     },
   });
   ```

3. **拉取结果请求 (`GET /job/{id}/result/markdown`)**
   ```javascript
   const resultRes = await fetch(`${baseURL}/job/${jobId}/result/${resultType}`, {
     headers: { 
       'Authorization': `Bearer ${apiKey}`,
       'Connection': 'close' // 禁用 Keep-Alive
     },
   });
   ```
